Git is the version control system behind virtually every software project in 2026. Whether you are a solo developer or part of a team of hundreds, the same core commands handle the vast majority of your daily work. This cheat sheet covers every command you need, organized by workflow, with copy-paste examples that use realistic branch names, file paths, and scenarios.

Repository Setup: init, clone, remote

Every Git workflow starts with either creating a new repository or cloning an existing one.

git init

Creates a new Git repository in the current directory by adding a hidden .git folder that stores all version history, branches, and configuration.

# Create and initialize a new project
mkdir my-api && cd my-api
git init

# Initialize with a specific default branch name
git init --initial-branch=main

# Initialize a bare repository (for servers)
git init --bare /srv/repos/my-api.git

After initializing, your first step should be creating a .gitignore file. The Gitignore Generator builds one for any tech stack in seconds.

git clone

Downloads an existing repository including its full history and sets up the remote connection automatically.

# Clone via HTTPS
git clone https://github.com/user/repo.git

# Clone via SSH (preferred for write access)
git clone git@github.com:user/repo.git

# Clone into a specific directory
git clone git@github.com:user/repo.git my-local-name

# Shallow clone (only last 10 commits, faster for large repos)
git clone --depth 10 https://github.com/user/repo.git

# Clone a specific branch
git clone --branch develop https://github.com/user/repo.git

git remote

Manages connections to remote repositories. A clone already sets up origin, but you may need additional remotes for forks or deploy targets.

# List remotes
git remote -v

# Add a new remote
git remote add upstream https://github.com/original/repo.git

# Change a remote URL (e.g., HTTPS to SSH)
git remote set-url origin git@github.com:user/repo.git

# Remove a remote
git remote remove upstream

# Rename a remote
git remote rename origin github

Branching: branch, checkout, switch

Branches let you work on features, fixes, and experiments in isolation. They are lightweight in Git because a branch is just a pointer to a commit.

git branch

# List local branches (* marks current branch)
git branch

# List all branches (local + remote)
git branch -a

# List branches with last commit info
git branch -v

# Create a new branch (does NOT switch to it)
git branch feature/search

# Create a branch from a specific commit
git branch hotfix/auth abc1234

# Rename the current branch
git branch -m new-name

# Rename a different branch
git branch -m old-name new-name

# Delete a merged branch
git branch -d feature/search

# Force-delete an unmerged branch
git branch -D experiment/broken-idea

# List branches merged into current branch
git branch --merged

# List branches NOT merged into current branch
git branch --no-merged

git switch (recommended) and git checkout

Git 2.23 introduced git switch specifically for changing branches. The older git checkout still works but does too many things, which leads to confusion.

# Switch to an existing branch
git switch main
git checkout main          # older equivalent

# Create and switch to a new branch
git switch -c feature/cart
git checkout -b feature/cart   # older equivalent

# Switch to a remote branch (creates local tracking branch)
git switch feature/api
# If origin/feature/api exists, Git sets up tracking automatically

# Discard changes in a file (checkout only)
git checkout -- src/app.js

# Restore a file from another branch
git checkout main -- config/settings.json

Tip: Use git switch for changing branches and git restore for discarding changes. This separation makes each command's intent clear. git checkout still does both, but mixing concerns leads to mistakes.

Staging and Committing: add, commit, status, diff

The staging area (index) is Git's holding zone between your working directory and the repository. You choose exactly which changes go into each commit.

git add

# Stage a specific file
git add src/cart.js

# Stage multiple files
git add src/cart.js src/api/cart-api.js tests/cart.test.js

# Stage all changes in current directory and subdirectories
git add .

# Stage all tracked files (skips new untracked files)
git add -u

# Stage changes interactively (choose hunks)
git add -p
# Git shows each change and asks: stage this hunk? [y/n/s/q]

# Stage specific parts of a file
git add -p src/cart.js

git commit

# Commit with a message
git commit -m "feat: add shopping cart component"

# Commit with a multi-line message
git commit -m "feat: add shopping cart" -m "Includes add/remove/update quantity functionality.
Closes #42."

# Stage all tracked changes AND commit (skip git add)
git commit -am "fix: correct price calculation rounding"

# Amend the last commit (change message or add files)
git add forgotten-file.js
git commit --amend -m "feat: add shopping cart with forgotten file"

# Amend without changing the message
git commit --amend --no-edit

# Create an empty commit (useful for triggering CI)
git commit --allow-empty -m "ci: trigger deploy pipeline"

git status and git diff

# Check current state
git status

# Short format status
git status -s

# Show unstaged changes (working dir vs staging)
git diff

# Show staged changes (staging vs last commit)
git diff --staged

# Show changes between two branches
git diff main..feature/cart

# Show changes in a specific file
git diff src/cart.js

# Show only file names that changed
git diff --name-only main..feature/cart

# Show a stat summary
git diff --stat main..feature/cart

Remote Operations: push, pull, fetch

git push

# Push current branch to its upstream
git push

# Push and set upstream tracking (first push of a new branch)
git push -u origin feature/cart

# Push a specific branch
git push origin main

# Push all branches
git push --all origin

# Push tags
git push --tags

# Force push (use with extreme care, only on your own branches)
git push --force-with-lease
# --force-with-lease is safer than --force: it fails if the remote
# has commits you haven't fetched yet

# Delete a remote branch
git push origin --delete feature/old-branch

git pull

# Pull and merge (default behavior)
git pull

# Pull and rebase instead of merge (cleaner history)
git pull --rebase

# Pull from a specific remote and branch
git pull origin develop

# Pull with autostash (stashes dirty working dir, pulls, pops stash)
git pull --rebase --autostash

git fetch

# Fetch all remotes
git fetch --all

# Fetch a specific remote
git fetch origin

# Fetch and prune deleted remote branches
git fetch --prune

# Fetch a specific branch
git fetch origin feature/api

fetch vs pull: git fetch downloads changes but does not touch your working directory. git pull fetches and then merges (or rebases). Fetching first lets you review incoming changes with git log origin/main before integrating them. Many teams prefer git fetch + explicit git merge or git rebase.

Merging and Rebasing: merge, rebase

git merge

# Merge a branch into your current branch
git switch main
git merge feature/cart

# Merge with a custom commit message
git merge feature/cart -m "Merge feature/cart: shopping cart v2"

# Merge without fast-forward (always create a merge commit)
git merge --no-ff feature/cart

# Abort a merge with conflicts
git merge --abort

# Continue after resolving conflicts
git add .
git merge --continue

git rebase

# Rebase current branch onto main
git switch feature/cart
git rebase main

# Interactive rebase: squash, reorder, edit commits
git rebase -i main
# In the editor, change "pick" to:
#   squash (s) - combine with previous commit
#   fixup (f) - combine and discard message
#   reword (r) - change commit message
#   edit (e)  - pause to amend the commit
#   drop (d)  - remove the commit

# Rebase the last 5 commits (rewrite recent history)
git rebase -i HEAD~5

# Abort a rebase
git rebase --abort

# Continue after resolving conflicts
git add .
git rebase --continue

# Skip the current commit during rebase
git rebase --skip
Golden Rule of Rebasing

Never rebase commits that have been pushed to a shared branch. Rebasing rewrites commit hashes. If others have based work on the old commits, their history diverges from the rewritten one, causing painful conflicts. Rebase only your local, unpushed work.

Stashing Changes: stash

Git stash shelves uncommitted changes so you can switch context without committing half-finished work.

# Stash all uncommitted changes
git stash

# Stash with a description
git stash push -m "WIP: cart total calculation"

# Stash including untracked files
git stash -u

# Stash including untracked AND ignored files
git stash -a

# List all stashes
git stash list
# Output: stash@{0}: On feature/cart: WIP: cart total calculation

# Apply the most recent stash (keeps stash in list)
git stash apply

# Apply and remove the most recent stash
git stash pop

# Apply a specific stash
git stash apply stash@{2}

# Show what a stash contains (diff)
git stash show -p stash@{0}

# Drop a specific stash
git stash drop stash@{1}

# Clear all stashes
git stash clear

# Create a branch from a stash
git stash branch feature/recover stash@{0}

History and Inspection: log, show, blame

git log

# Default log
git log

# One-line format (compact)
git log --oneline

# Show a graph of branches
git log --oneline --graph --all

# Log with diff of changes
git log -p

# Limit to last N commits
git log -5

# Log commits by a specific author
git log --author="Jane"

# Log commits since a date
git log --since="2026-01-01"

# Log commits that changed a specific file
git log -- src/cart.js

# Search commit messages with grep
git log --grep="fix: payment"

# Show merge commits only
git log --merges

# Custom format
git log --pretty=format:"%h %ad | %s [%an]" --date=short

git show and git blame

# Show a specific commit's details and diff
git show abc1234

# Show a file from a specific commit
git show abc1234:src/cart.js

# Blame: who changed each line of a file
git blame src/cart.js

# Blame a specific range of lines
git blame -L 10,25 src/cart.js

# Blame ignoring whitespace changes
git blame -w src/cart.js

Undoing Changes: reset, revert, restore, cherry-pick

git reset

# Soft reset: undo commit, keep changes staged
git reset --soft HEAD~1

# Mixed reset (default): undo commit, unstage changes
git reset HEAD~1

# Hard reset: undo commit, discard ALL changes
git reset --hard HEAD~1

# Reset to a specific commit
git reset --hard abc1234

# Unstage a file (keep working dir changes)
git reset HEAD src/cart.js
# Modern alternative:
git restore --staged src/cart.js

git revert

# Revert a commit (creates a new "undo" commit)
git revert abc1234

# Revert without auto-committing
git revert --no-commit abc1234

# Revert a merge commit (specify which parent to keep)
git revert -m 1 abc1234

reset vs revert: Use reset for local, unpushed commits you want to erase. Use revert for pushed commits on shared branches because it creates a new commit instead of rewriting history.

git restore

# Discard working directory changes in a file
git restore src/cart.js

# Restore a file from a specific commit
git restore --source abc1234 src/cart.js

# Unstage a file
git restore --staged src/cart.js

git cherry-pick

# Apply a specific commit to the current branch
git cherry-pick abc1234

# Cherry-pick without auto-committing
git cherry-pick --no-commit abc1234

# Cherry-pick a range of commits
git cherry-pick abc1234..def5678

# Continue after resolving conflicts
git cherry-pick --continue

# Abort the cherry-pick
git cherry-pick --abort

Advanced: bisect, reflog, worktree, submodule

git bisect

Binary search through commit history to find the exact commit that introduced a bug. For a repository with 1,000 commits, bisect finds the culprit in about 10 steps.

# Start bisecting
git bisect start

# Mark current state as broken
git bisect bad

# Mark a known good commit
git bisect good v1.2.0

# Git checks out a midpoint commit. Test it, then:
git bisect good   # if this commit works
git bisect bad    # if this commit is broken

# Repeat until Git identifies the first bad commit

# Automate with a test script
git bisect run npm test

# When done, return to original branch
git bisect reset

git reflog

The reflog is your safety net. It records every position HEAD has pointed to, including commits that are no longer reachable from any branch. Accidentally deleted a branch or ran a bad reset? The reflog still has the reference for at least 90 days.

# View the reflog
git reflog

# Output example:
# abc1234 HEAD@{0}: reset: moving to HEAD~3
# def5678 HEAD@{1}: commit: feat: add payment integration
# ghi9012 HEAD@{2}: checkout: moving from main to feature/pay

# Recover a lost commit
git checkout abc1234
# Or create a branch pointing to it
git branch recovered abc1234

# See reflog for a specific branch
git reflog show feature/cart

git worktree

Worktrees let you check out multiple branches simultaneously in separate directories. No more stashing or committing half-done work just to look at another branch.

# Add a worktree for a different branch
git worktree add ../hotfix-dir hotfix/payment

# Now you have two directories:
#   /project (main branch)
#   /hotfix-dir (hotfix/payment branch)

# Add a worktree with a new branch
git worktree add -b feature/new-ui ../new-ui-dir

# List all worktrees
git worktree list

# Remove a worktree
git worktree remove ../hotfix-dir

# Prune stale worktree info
git worktree prune

git submodule

# Add a submodule
git submodule add https://github.com/lib/utils.git vendor/utils

# Clone a repo that contains submodules
git clone --recurse-submodules https://github.com/user/repo.git

# Initialize submodules after a regular clone
git submodule update --init --recursive

# Update all submodules to latest
git submodule update --remote

# Remove a submodule
git submodule deinit vendor/utils
git rm vendor/utils

git tag

# Create a lightweight tag
git tag v1.0.0

# Create an annotated tag (recommended for releases)
git tag -a v1.0.0 -m "Release version 1.0.0"

# Tag a specific commit
git tag -a v0.9.0 abc1234 -m "Beta release"

# List all tags
git tag -l

# List tags matching a pattern
git tag -l "v1.*"

# Push a specific tag
git push origin v1.0.0

# Push all tags
git push --tags

# Delete a local tag
git tag -d v1.0.0

# Delete a remote tag
git push origin --delete v1.0.0

git clean

# Preview what would be deleted (dry run)
git clean -n

# Remove untracked files
git clean -f

# Remove untracked files AND directories
git clean -fd

# Remove untracked AND ignored files
git clean -fdx

Real-World Workflows

Feature Branch Workflow

The standard for most teams. Each feature gets its own branch, and changes merge via pull requests.

# 1. Start from updated main
git switch main
git pull --rebase

# 2. Create feature branch
git switch -c feature/user-dashboard

# 3. Work on the feature (multiple focused commits)
git add src/dashboard/
git commit -m "feat: scaffold dashboard layout"
git add src/api/dashboard.js
git commit -m "feat: add dashboard API integration"
git add tests/dashboard.test.js
git commit -m "test: add dashboard component tests"

# 4. Stay up to date with main
git fetch origin
git rebase origin/main

# 5. Push and open PR
git push -u origin feature/user-dashboard

# 6. After PR approval and merge, clean up
git switch main
git pull
git branch -d feature/user-dashboard

Hotfix Workflow

# 1. Branch from latest main
git switch main
git pull
git switch -c hotfix/payment-overflow

# 2. Fix the bug
git add src/payment.js
git commit -m "fix: prevent integer overflow in payment total"

# 3. Push and merge via expedited PR
git push -u origin hotfix/payment-overflow

# 4. After merge, tag the release
git switch main
git pull
git tag -a v1.2.1 -m "Hotfix: payment overflow"
git push --tags

Interactive Rebase Before Merge

Clean up messy commit history before opening a pull request.

# 1. Rebase onto latest main
git switch feature/search
git fetch origin
git rebase -i origin/main

# 2. In the interactive editor, squash fixup commits:
# pick abc1234 feat: add search component
# squash def5678 fix: typo in search
# squash ghi9012 fix: search styling
# pick jkl3456 test: add search tests

# 3. Force push (safe because it's your branch)
git push --force-with-lease

Working with Forks

# 1. Clone your fork
git clone git@github.com:you/repo.git
cd repo

# 2. Add upstream remote
git remote add upstream https://github.com/original/repo.git

# 3. Sync fork with upstream
git fetch upstream
git switch main
git merge upstream/main
git push origin main

# 4. Create feature branch and work
git switch -c feature/my-contribution
# ... make changes, commit, push to your fork ...
git push -u origin feature/my-contribution
# Open PR from your fork to upstream

Frequently Asked Questions

Git merge creates a new merge commit that joins two branches, preserving the full history of both. Git rebase replays your commits on top of another branch, producing a linear history without merge commits. Use merge on shared branches where you want a complete record of when integration happened. Use rebase on local feature branches to keep history clean before opening a pull request. The golden rule: never rebase commits that have already been pushed to a shared remote, because rewriting public history forces everyone else to reconcile the divergence.

Use git reset --soft HEAD~1 to undo the commit while keeping all changes staged. Use git reset --mixed HEAD~1 (or simply git reset HEAD~1) to undo the commit and unstage the changes, leaving them in your working directory. Use git reset --hard HEAD~1 to discard the commit and all changes permanently. If the commit has already been pushed to a remote, use git revert HEAD instead. Revert creates a new commit that undoes the changes without rewriting history, which is safe for shared branches.

Git stash temporarily shelves uncommitted changes so you can work on something else without committing half-finished code. Run git stash to save your work, do whatever you need on another branch, then run git stash pop to restore your changes. Use git stash list to view all stashes, git stash apply stash@{n} to apply a specific one without removing it, and git stash drop stash@{n} to delete one. Common use cases include switching branches mid-task, pulling remote changes on a dirty working directory, and testing something on a clean slate.

Git fetch downloads commits, branches, and tags from the remote but does not touch your working directory or current branch. It updates remote-tracking branches like origin/main so you can review what changed before integrating. Git pull is shorthand for git fetch followed by git merge (or git rebase if configured). It downloads and immediately integrates remote changes. Using fetch first is safer because you can inspect incoming commits with git log origin/main or git diff before merging. Many teams prefer git fetch plus an explicit git merge or git rebase for more control.

Git bisect performs a binary search through your commit history to find the exact commit that introduced a bug. Start with git bisect start, then mark the current broken state with git bisect bad and a known working commit with git bisect good <hash>. Git checks out a commit halfway between the two. You test it and mark it good or bad. Git narrows the range and checks out the next midpoint. This repeats until the offending commit is identified. For 1,000 commits, bisect needs roughly 10 steps instead of testing each one. Run git bisect reset when finished to return to your original branch. You can also automate the process with git bisect run <test-script>.

Git reflog is a local log of every position HEAD has pointed to, including commits that are no longer reachable from any branch. If you accidentally delete a branch, hard reset past a commit, or lose work after a bad rebase, the reflog still has a reference to those commits for at least 90 days. Run git reflog to see the history, find the hash of the lost commit, and recover it with git checkout <hash> or git branch recovered <hash>. The reflog is local only and is not shared with remotes, so it only works on the machine where the action happened.
