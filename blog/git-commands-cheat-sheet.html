Basic Commands

Every Git workflow starts with the same fundamentals: initializing a repository, staging files, and committing changes. These commands form the backbone of version control, and you will use them dozens of times every day.

Creating and Cloning Repositories

bash — Repository setup
# Initialize a new Git repository in the current directory
git init

# Initialize with a specific default branch name
git init --initial-branch=main

# Clone a remote repository
git clone https://github.com/user/repo.git

# Clone into a specific directory
git clone https://github.com/user/repo.git my-project

# Clone only the latest commit (shallow clone, faster)
git clone --depth 1 https://github.com/user/repo.git

# Clone a specific branch
git clone --branch develop https://github.com/user/repo.git

Staging and Committing

bash — The add-commit cycle
# Check the status of your working directory
git status

# Stage a specific file
git add index.html

# Stage multiple files
git add src/app.js src/utils.js

# Stage all changes in the current directory
git add .

# Stage all tracked files (skip untracked)
git add -u

# Interactive staging (choose hunks)
git add -p

# Commit staged changes with a message
git commit -m "Add user authentication module"

# Stage all tracked files and commit in one step
git commit -am "Fix login redirect bug"

# Amend the last commit message
git commit --amend -m "Add user authentication with OAuth"

# Amend the last commit by adding more files (no message change)
git add forgotten-file.js
git commit --amend --no-edit
Commit Message Convention

Use the imperative mood: "Add feature" not "Added feature." Keep the subject line under 50 characters. Add a blank line before detailed descriptions. Prefix with a type for consistency: feat:, fix:, docs:, refactor:, test:, chore:.

Viewing File Changes

bash — Inspecting changes
# Show unstaged changes
git diff

# Show staged changes (what will be committed)
git diff --staged

# Show changes for a specific file
git diff src/app.js

# Show a summary of changed files
git diff --stat

Branching

Branches are lightweight pointers to commits. They let you work on features, fixes, and experiments in isolation without affecting the main codebase. Git branches are cheap to create and fast to switch between because they are just 41-byte files containing a commit hash.

bash — Branch operations
# List all local branches
git branch

# List all branches (local and remote)
git branch -a

# List branches with last commit info
git branch -v

# List branches that have been merged into current
git branch --merged

# List branches NOT yet merged into current
git branch --no-merged

# Create a new branch
git branch feature/user-auth

# Create and switch to a new branch
git checkout -b feature/user-auth

# Create and switch (modern syntax)
git switch -c feature/user-auth

# Switch to an existing branch
git checkout main
git switch main

# Rename the current branch
git branch -m new-name

# Rename a specific branch
git branch -m old-name new-name

# Delete a branch (safe, only if merged)
git branch -d feature/user-auth

# Force delete a branch (even if unmerged)
git branch -D feature/user-auth

# Delete a remote branch
git push origin --delete feature/user-auth
Branch Naming Convention

Use prefixes to categorize branches: feature/ for new features, fix/ or bugfix/ for bug fixes, hotfix/ for urgent production fixes, release/ for release prep, and chore/ for maintenance tasks. Example: feature/user-auth-oauth2.

Merging and Rebasing

Merging and rebasing are two strategies for integrating changes from one branch into another. Merge preserves the full branch history by creating a merge commit. Rebase rewrites history to create a clean, linear sequence of commits. Both have their place in a professional workflow.

Merging

bash — Merge operations
# Merge a branch into the current branch
git merge feature/user-auth

# Merge with a custom commit message
git merge feature/user-auth -m "Merge user auth feature"

# Merge without fast-forward (always create merge commit)
git merge --no-ff feature/user-auth

# Abort a merge in progress (conflict resolution)
git merge --abort

# Continue after resolving merge conflicts
git add .
git merge --continue

# Squash merge (combine all branch commits into one)
git merge --squash feature/user-auth
git commit -m "Add user authentication (squashed)"

Rebasing

bash — Rebase operations
# Rebase current branch onto main
git rebase main

# Interactive rebase (edit, squash, reorder last 5 commits)
git rebase -i HEAD~5

# Interactive rebase onto main
git rebase -i main

# Continue rebase after resolving conflicts
git rebase --continue

# Skip the current conflicting commit
git rebase --skip

# Abort a rebase in progress
git rebase --abort

# Rebase and automatically squash fixup commits
git rebase -i --autosquash main
The Golden Rule of Rebasing

Never rebase commits that have been pushed to a shared remote branch. Rebasing rewrites commit hashes. If others have based work on those commits, their history diverges and they will need to force-reconcile. Rebase local branches before merging. Merge shared branches.

Interactive Rebase Actions

Action Short What It Does
pick p Use the commit as-is
reword r Use commit but edit the message
edit e Pause at commit for amending
squash s Combine with previous commit, edit message
fixup f Combine with previous commit, discard message
drop d Remove the commit entirely

Remote Repositories

Remote repositories are hosted copies of your project on services like GitHub, GitLab, or Bitbucket. Git tracks these as named remotes, with origin being the default name for the repository you cloned from. You can have multiple remotes for the same project.

bash — Remote operations
# List all remotes
git remote -v

# Add a new remote
git remote add origin https://github.com/user/repo.git

# Add a second remote (for forks)
git remote add upstream https://github.com/original/repo.git

# Rename a remote
git remote rename origin github

# Remove a remote
git remote remove upstream

# Show detailed info about a remote
git remote show origin

# Fetch changes from a remote (does not merge)
git fetch origin

# Fetch from all remotes
git fetch --all

# Fetch and prune deleted remote branches
git fetch --prune

# Pull changes (fetch + merge)
git pull origin main

# Pull with rebase instead of merge
git pull --rebase origin main

# Push to a remote branch
git push origin main

# Push and set upstream tracking
git push -u origin feature/user-auth

# Push all branches
git push --all origin

# Force push (CAUTION: overwrites remote history)
git push --force-with-lease origin feature/user-auth

# Push tags to remote
git push origin --tags
force-with-lease vs force

Always prefer --force-with-lease over --force. It checks that the remote branch has not been updated by someone else since your last fetch. If it has, the push is rejected, preventing you from accidentally overwriting a teammate's work.

Stashing

Stashing saves your uncommitted changes to a temporary stack so you can switch contexts without committing half-finished work. It captures both staged and unstaged modifications, reverting your working directory to the last commit.

bash — Stash operations
# Stash current changes
git stash

# Stash with a descriptive message
git stash push -m "WIP: user auth form validation"

# Stash including untracked files
git stash -u

# Stash only staged changes
git stash push --staged

# List all stashes
git stash list

# Show contents of the latest stash
git stash show

# Show full diff of the latest stash
git stash show -p

# Apply the latest stash (keep it in the stack)
git stash apply

# Apply and remove the latest stash
git stash pop

# Apply a specific stash by index
git stash apply stash@{2}

# Drop a specific stash
git stash drop stash@{0}

# Clear all stashes (CAUTION: cannot be undone)
git stash clear

# Create a new branch from a stash
git stash branch new-feature stash@{0}

Log and Diff

Git's log and diff commands are your primary tools for understanding what happened in a repository, when it happened, and who did it. Mastering the formatting options turns raw commit history into actionable intelligence.

Git Log

bash — Log formats and filters
# Basic log
git log

# Compact one-line format
git log --oneline

# Show last 10 commits
git log -10

# Log with graph visualization
git log --oneline --graph --all --decorate

# Log with file change stats
git log --stat

# Log with full diff for each commit
git log -p

# Search commits by message
git log --grep="fix login"

# Search commits by author
git log --author="Jane"

# Search commits by date range
git log --after="2026-01-01" --before="2026-02-01"

# Log for a specific file
git log -- src/auth.js

# Log for a specific file (follow renames)
git log --follow -- src/auth.js

# Custom format
git log --pretty=format:"%h %an %ar %s"

# Show commits on branch-a that are not on main
git log main..branch-a

# Show who last modified each line
git blame src/auth.js

# Blame with line range
git blame -L 10,20 src/auth.js

Git Diff

bash — Diff operations
# Diff between working directory and staging area
git diff

# Diff between staging area and last commit
git diff --staged

# Diff between two branches
git diff main..feature/user-auth

# Diff between two commits
git diff abc1234 def5678

# Diff showing only file names
git diff --name-only main..feature/user-auth

# Diff showing file names with status (A/M/D)
git diff --name-status main..feature/user-auth

# Word-level diff (instead of line-level)
git diff --word-diff

# Diff with statistics summary
git diff --stat main..feature/user-auth

Undoing Changes

Making mistakes is inevitable. Git provides multiple levels of undo, from discarding unstaged changes to rewriting public history. Choosing the right undo command depends on whether the change is uncommitted, committed locally, or already pushed to a remote.

Unstaged and Staged Changes

bash — Undo uncommitted changes
# Discard changes in a specific file (revert to last commit)
git checkout -- src/app.js

# Modern syntax for discarding changes
git restore src/app.js

# Discard ALL unstaged changes
git restore .

# Unstage a file (keep changes in working directory)
git reset HEAD src/app.js

# Modern syntax for unstaging
git restore --staged src/app.js

# Unstage everything
git restore --staged .

Undoing Commits

bash — Undo commits
# Undo the last commit, keep changes staged
git reset --soft HEAD~1

# Undo the last commit, keep changes unstaged
git reset --mixed HEAD~1

# Undo the last commit, discard ALL changes (DESTRUCTIVE)
git reset --hard HEAD~1

# Undo the last 3 commits
git reset --soft HEAD~3

# Revert a commit by creating a NEW undo commit (safe for shared branches)
git revert HEAD

# Revert a specific commit
git revert abc1234

# Revert a merge commit (specify parent)
git revert -m 1 abc1234

# Revert without auto-committing (stage changes only)
git revert --no-commit abc1234
reset vs revert

git reset rewrites history and is safe only for local, unpushed commits. git revert creates a new commit that undoes changes and is safe for shared branches. If the commit has been pushed, always use git revert.

Emergency Recovery with Reflog

bash — Reflog recovery
# View the reflog (every HEAD movement)
git reflog

# Show reflog for a specific branch
git reflog show feature/user-auth

# Recover a commit after accidental hard reset
git reflog
# Find the commit hash you want to recover to
git reset --hard abc1234

# Recover a deleted branch
git reflog
git checkout -b recovered-branch abc1234
Reflog Is Your Safety Net

The reflog records every change to HEAD for at least 30 days by default. Even after git reset --hard, you can recover lost commits through the reflog. It is your last line of defense against accidental data loss.

Tags

Tags mark specific commits as important, typically for release versions. Unlike branches, tags do not move when new commits are made. Git supports lightweight tags (just a name pointing to a commit) and annotated tags (stored as full objects with metadata, a message, and an optional GPG signature).

bash — Tag operations
# List all tags
git tag

# List tags matching a pattern
git tag -l "v1.*"

# Create a lightweight tag
git tag v1.0.0

# Create an annotated tag (recommended for releases)
git tag -a v1.0.0 -m "Release version 1.0.0"

# Tag a specific commit
git tag -a v1.0.0 abc1234 -m "Release version 1.0.0"

# Show tag details
git show v1.0.0

# Push a specific tag to remote
git push origin v1.0.0

# Push all tags
git push origin --tags

# Delete a local tag
git tag -d v1.0.0

# Delete a remote tag
git push origin --delete v1.0.0

# Checkout a specific tag (detached HEAD)
git checkout v1.0.0

# Create a branch from a tag
git checkout -b hotfix/v1.0.1 v1.0.0

Configuration

Git configuration has three levels: system (all users), global (current user), and local (current repository). Local overrides global, which overrides system. Setting up your configuration correctly prevents issues with commit attribution, line endings, and merge behavior.

bash — Essential configuration
# Set your identity
git config --global user.name "Jane Developer"
git config --global user.email "jane@example.com"

# Set default branch name for new repos
git config --global init.defaultBranch main

# Set default editor
git config --global core.editor "code --wait"

# Enable color output
git config --global color.ui auto

# Set pull strategy to rebase
git config --global pull.rebase true

# Auto-prune on fetch
git config --global fetch.prune true

# Set merge conflict style (shows base version too)
git config --global merge.conflictstyle diff3

# Handle line endings (macOS/Linux)
git config --global core.autocrlf input

# Handle line endings (Windows)
git config --global core.autocrlf true

# List all config settings
git config --list

# List with origin (shows which file each setting comes from)
git config --list --show-origin

# View a specific setting
git config user.email

# Unset a config value
git config --global --unset user.email

Useful Git Aliases

bash — Time-saving aliases
# Set up common aliases
git config --global alias.st status
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.lg "log --oneline --graph --all --decorate"
git config --global alias.last "log -1 HEAD --stat"
git config --global alias.unstage "restore --staged"
git config --global alias.amend "commit --amend --no-edit"
git config --global alias.undo "reset --soft HEAD~1"

# Now you can use:
git st          # instead of git status
git co main     # instead of git checkout main
git lg          # pretty graph log
git undo        # undo last commit, keep changes

Advanced Tips

These commands handle the situations that basic tutorials do not cover: isolating bugs, applying specific commits across branches, cleaning up history, and working with multiple working trees simultaneously.

Cherry-Pick

bash — Cherry-pick operations
# Apply a specific commit to the current branch
git cherry-pick abc1234

# Cherry-pick without auto-committing
git cherry-pick --no-commit abc1234

# Cherry-pick a range of commits
git cherry-pick abc1234..def5678

# Abort a cherry-pick in progress
git cherry-pick --abort

Bisect (Binary Search for Bugs)

bash — Git bisect
# Start bisecting
git bisect start

# Mark the current commit as bad (has the bug)
git bisect bad

# Mark a known good commit
git bisect good v1.0.0

# Git checks out a middle commit. Test it, then:
git bisect good    # if the bug is NOT present
git bisect bad     # if the bug IS present

# Repeat until Git identifies the exact commit

# End bisect session
git bisect reset

# Automated bisect with a test script
git bisect start HEAD v1.0.0
git bisect run npm test

Worktree (Multiple Working Directories)

bash — Git worktree
# Add a new worktree for a branch
git worktree add ../hotfix-branch hotfix/critical-fix

# Add a worktree with a new branch
git worktree add -b feature/new ../feature-work main

# List all worktrees
git worktree list

# Remove a worktree
git worktree remove ../hotfix-branch

# Prune stale worktree references
git worktree prune

Clean and .gitignore

bash — Cleaning untracked files
# Show what would be deleted (dry run)
git clean -n

# Delete untracked files
git clean -f

# Delete untracked files and directories
git clean -fd

# Delete untracked and ignored files
git clean -fdx

# Check which files are ignored
git status --ignored

# Verify .gitignore rules against a path
git check-ignore -v src/secrets.env

Frequently Asked Questions

git merge creates a new merge commit that combines two branches, preserving the full history of both branches as they were. git rebase rewrites history by moving your branch's commits on top of another branch, creating a linear history. Use merge for shared branches like main where you want a clear record of when features were integrated. Use rebase for local feature branches before merging to keep the commit history clean. The rule of thumb: never rebase commits that have been pushed to a shared remote branch, because it rewrites commit hashes and forces everyone else to reconcile the diverged history.

Use git reset --soft HEAD~1 to undo the last commit while keeping all changes staged and ready to commit again. Use git reset --mixed HEAD~1 (or just git reset HEAD~1) to undo the commit and unstage the changes, but keep the files modified in your working directory. Use git reset --hard HEAD~1 to undo the commit and discard all changes permanently. If the commit has already been pushed to a remote, use git revert HEAD instead, which creates a new commit that undoes the changes without rewriting history. This is safe for shared branches because it does not alter existing commit hashes.

git stash temporarily saves your uncommitted changes (both staged and unstaged) and reverts your working directory to a clean state. Use it when you need to switch branches but are not ready to commit your current work, when you need to pull remote changes but have local modifications, or when you want to test something on a clean codebase without losing progress. Run git stash to save changes, do your other work, then run git stash pop to restore them. You can have multiple stashes and list them with git stash list. Use git stash apply to restore without removing from the stash stack, and git stash drop to remove a specific stash entry.

When Git cannot automatically merge changes, it marks the conflicting sections in the affected files with conflict markers: <<<<<<< HEAD shows your current branch's version, ======= separates the two versions, and >>>>>>> branch-name shows the incoming branch's version. To resolve: 1) Open each conflicted file and edit it to keep the code you want, removing all conflict markers. 2) Stage the resolved files with git add. 3) Complete the merge with git commit. You can use git status to see which files have conflicts. Tools like git mergetool, VS Code's built-in merge editor, or the ANIMA Diff Checker can help visualize and resolve conflicts. To abort a merge in progress, use git merge --abort.

git fetch downloads new commits and branches from the remote repository but does not modify your working directory or current branch. It updates your remote-tracking branches (like origin/main) so you can review changes before integrating them. git pull is essentially git fetch followed by git merge. It downloads remote changes and immediately merges them into your current branch. Many developers prefer fetch plus merge (or fetch plus rebase) as two separate steps because it gives you a chance to inspect incoming changes with git log origin/main before integrating them. You can set git pull to use rebase instead of merge by default with git config --global pull.rebase true.

Use git push origin --delete branch-name to delete a branch on the remote repository. To delete the local copy of the branch, use git branch -d branch-name for a safe delete (only works if the branch has been fully merged) or git branch -D branch-name to force delete regardless of merge status. After deleting remote branches, other team members should run git fetch --prune to clean up their local references to the deleted remote branch. You can also configure automatic pruning with git config --global fetch.prune true so stale remote-tracking branches are removed every time you fetch.
